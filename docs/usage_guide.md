# NAMU 사용설명서 — 설치 후 첫 하루

> 📅 2026-07-09 · hp 기기에서 실제 타 프로젝트(설치형·원격 marketplace)에 설치해 실작업 1건을 완주한 실측(#25) 채록을 사용자 관점으로 정리한 문서.

이 문서는 [`install_guide.md`](install_guide.md)를 따라 **설치를 이미 마친 사용자**가 NAMU를 실제로 쓰기 시작하는 첫 하루를 다룬다. 설치 절차 자체는 다루지 않는다. 독자 전제는 install_guide와 같다(Claude Code 또는 agy를 쓸 줄 아는 사람).

## 1. 설치가 성공했는지 확인하는 법

**첫 세션은 조용한 게 정상이다.** 설치 직후 세션을 열면 NAMU의 존재감이 전혀 없다 — 환영 메시지도, 브리핑도 안 뜬다. 이유는 두 가지가 겹쳐서다.

- 세션 브리핑은 **보여줄 기억(진행 중 task·교훈)이 있어야만** 만들어진다. 신규 환경은 둘 다 0건이라 브리핑 훅이 설계상 아무것도 출력하지 않는다.
- 브리핑이 있더라도 Claude Code 2.1.0+는 SessionStart 훅 출력을 **화면에 보여주지 않고 AI의 컨텍스트에만 주입**한다. 사용자 눈에는 원래 안 보인다.

그러니 "아무것도 안 뜨는데 설치 실패인가?"라고 오인하지 말고, 다음 두 가지로 확인하라 (실측 확인된 방법).

1. **`/mcp`** — 서버 목록에 `plugin:namu:namu-memory · ✔ connected · 3 tools`가 보이면 메모리 코어가 살아 있는 것이다.
2. **AI에게 직접 묻기** — "namu 메모리에서 recall 해봐"라고 하면 AI가 `namu_recall` 도구를 호출한다. 신규 환경에서는 "저장된 기억이 비어 있다"는 답이 나오는데, **이 '비어 있음'이 곧 정상 동작의 증거**다(도구 호출 자체가 실패하면 설치 문제).

## 2. 기억은 어디에 쌓이는가

NAMU는 저장소를 **이원화**한다 — 교훈(learnings)·검색 캐시(db)와 작업 상태(tasks)는 서로 다른 곳에 쌓인다.

**교훈·db — `~/.namu`(고정 경로, namu-35)**. 지정하거나 바꿀 수 있는 환경변수가 없다. 실측으로 확인된 생성 시점은 다음과 같다.

- `~/.namu/db/` — **MCP 서버가 처음 부팅될 때** 자동 생성된다(`claude mcp list`만 해도 생긴다). 교훈을 처음 기록할 때가 아니다.
- `~/.namu/memory/learnings.yaml` — **첫 교훈이 기록될 때** 생성된다.

이 둘은 프로젝트와 무관한 지식이라 기기 전역(글로벌)으로 공유되도록 설계됐다 — NAMU 자체를 개발하며 얻은 교훈도 이 풀에 함께 쌓인다(namu-35: 별도 카테고리로 분리하지 않는다).

**작업 상태(tasks) — 개인 풀 `~/.namu/tasks/<basename(프로젝트 폴더)>/`(namu-34)**. `/namu-task`를 돌리면 현재 프로젝트 폴더의 이름(basename) 하나를 기준으로 이 개인 풀 아래에 만들어진다 — 4절 참고.

## 3. 첫 작업 돌리기 — `/namu-task`

NAMU의 본체험은 `/namu-task`로 실작업 1건을 돌려보는 것이다. 실측(영화관 썸네일 버그 리뷰)에서 확인된 흐름:

1. **작업 정의** — 슬러그와 목적·완료조건을 AI가 초안하고 **사용자 확인**을 받아 확정한다.
2. **코딩 분담** — 코더 워커(서브에이전트)가 실제 수정을 수행한다. 워커 호출 전에도 사용자 확인을 거친다.
3. **검수** — 리뷰어 워커가 pass/fail 판정. fail이면 자동 재실행하지 않고 사용자에게 묻는다.
4. **교훈 기록** — 일반화할 교훈이 있으면 `namu_record`로 저장된다(`~/.namu/memory/learnings.yaml`에 reason 포함으로 남는 걸 확인할 수 있다).
5. **마무리** — 커밋·push 같은 되돌리기 어려운 동작은 사용자 승인을 기다린다.

실측에서는 이 전 사이클(작업 생성 → 버그 3건 수정 → 검수 PASS → 교훈 기록 → 사용자 승인 → 커밋·push)이 설치형 환경에서 끝까지 완주됐다.

스킬 이름이 안 잡히면 `/`를 눌러 자동완성 목록에서 `namu`로 시작하는 항목을 찾아라(설치형에서는 플러그인 네임스페이스가 붙어 이름이 다르게 보일 수 있다).

## 4. 알아둘 현 버전 동작 (실측 채록)

첫 하루에 마주칠 수 있는, 버그라기보다 "지금은 이렇게 동작한다"들이다.

- **작업 상태 파일은 개인 풀에 생긴다 (namu-34, 의도된 설계)** — `/namu-task`를 돌리면 `tasks/<슬러그>/`(task.md·context·log)가 그 프로젝트 폴더 안이 아니라 **`~/.namu/tasks/<basename(프로젝트 폴더)>/`**에 만들어진다. 교훈·db(`~/.namu/memory`·`~/.namu/db`, 프로젝트 독립 지식)와 tasks(프로젝트 종속 상태이지만 저장 위치는 개인 풀)는 산출 기준은 다르지만 물리적으로는 같은 `~/.namu` 계열에 모인다 — 공개 repo에 개인 작업 기록이 노출되는 걸 막고, 프로젝트 자체의 git 유무와 무관하게 PC 간 자동 공유가 되도록 하기 위해서다. 원하는 프로젝트 git으로 tasks를 함께 추적하고 싶은 경우는 지원하지 않는다(전부 개인 풀로 통합).
- **machine 도장** — 환경변수 `NAMU_MACHINE`을 설정하지 않아도, 현 버전은 호스트명(`platform.node()`)으로 자동 폴백해 작업 상태 파일에 그 값이 찍힌다(우선순위: `NAMU_MACHINE` 환경변수 → 호스트명 → 그래도 비면 `unknown`). `unknown`이 찍히는 건 호스트명조차 얻을 수 없는 드문 환경뿐이다. 그래도 `NAMU_MACHINE` 명시 설정은 권장(install_guide 2절) — 멀티 PC에서 라벨을 사람이 읽기 쉽게 고정하고 싶을 때 유용하다.
- **브리핑은 다음 세션부터** — 교훈·진행 task가 생긴 뒤 **새로 여는 세션**에서 AI 컨텍스트로 주입된다. 화면에는 여전히 안 보이므로, 확인하고 싶으면 AI에게 "지금 진행 중인 task 뭐라고 알고 있어?"라고 물어보라.

## 5. statusLine — 플러그인에 동봉되지만 수동 설정이 필요함

`[모델] 폴더 | 📌 진행 task | 컨텍스트%` 형태의 statusLine 스크립트가 `namu-plugin/scripts/namu_statusline.py`로 플러그인에 동봉된다(namu-26부터). 다만 **Claude Code 플러그인 매니페스트(`plugin.json`)는 화면 하단 statusLine을 자동 등록하는 메커니즘을 제공하지 않는다**(조사 결과 — `settings.json`의 `subagentStatusLine`은 서브에이전트 패널 행 커스터마이즈용 별개 기능이며, 메인 statusLine이 아니다). 그러므로 **설치형 사용자도 여전히 수동 연결이 필요**하다:

1. 설치본 경로를 찾는다 — 원격 설치라면 `~/.claude/plugins/cache/<marketplace>/<plugin>/<버전>/scripts/namu_statusline.py` (install_guide 3절 "원격 캐시" 참고).
2. 자신의 Claude Code `statusLine` 설정(`~/.claude/settings.json` 또는 프로젝트 `.claude/settings.json`)에 위 절대경로를 `command`로 지정한다.

참고로 각 칸의 `?`는 "값 없음" 폴백이다(예: 세션 초반의 컨텍스트 사용률). repo를 clone해 개발 중이면 대신 repo 루트 `scripts/namu_statusline.py`(동일 내용의 원본)를 가리켜도 된다.

## 6. 기억의 원격 백업·멀티 PC 동기화 (선택 기능, 0.1.11부터)

`~/.namu`에 쌓이는 기억(교훈)은 기본적으로 **그 기기의 로컬 폴더일 뿐**이다 — 디스크가 죽으면 기억도 죽고, PC를 두 대 이상 쓰면 기억이 기기마다 따로 쌓여 합쳐지지 않는다. 이 절은 그 한계를 없애는 선택적 동기화 기능(MCP 도구 `namu_sync_setup`)을 설명한다. **명시적으로 켜지 않으면 위 한계가 그대로 적용된다** — `~/.namu/.namu_sync` 마커 파일이 있어야만 동작한다.

### 켜는 법

1. GitHub 등에 **비공개 원격 저장소**를 하나 준비한다. "준비"의 범위는 딱 세 가지다:
   - **계정** — GitHub 기준으로 설명하지만, GitLab·사내 git 서버·NAS의 bare 저장소 등 git URL이 나오는 곳이면 어디든 된다.
   - **빈 비공개 저장소 생성** — New repository → 이름 자유(예: `namu-memory`) → **Private** 선택 → README 등 초기화 옵션 없이(빈 저장소 권장) 생성. 비공개여야 하는 이유: 교훈에는 작업한 프로젝트의 내용·판단·실수가 그대로 담긴다. GitHub은 비공개 저장소도 무료다.
   - **그 기기의 git 인증** — push 시 비밀번호를 묻지 않는 상태여야 한다. `gh auth login`(가장 간단) 또는 SSH 키 등록으로 준비하고, `gh auth status`로 확인한다. 그 기기에서 push해 본 적이 있으면 대개 이미 돼 있다.

   여기까지가 사용자 몫이고, 저장소 URL을 넘긴 순간부터는 NAMU가 로컬 쪽 git 연결(git init·remote·gitignore/gitattributes·초기 커밋·push)을 전부 대신한다.
2. AI에게 "`namu_sync_setup`을 `<원격 URL>`로 실행해줘"라고 자연어로 부탁한다. **한 번이면 끝**이다.

### 켠 뒤 동작 방식

- `namu_record`로 교훈을 기록할 때마다 **자동 commit + push**된다. push 실패는 record 결과에 영향을 주지 않는다(기록 자체는 항상 먼저 성공한다).
- Claude Code·agy 세션을 새로 열 때마다(양쪽 훅 공통) **자동 pull**로 다른 PC의 최신 기억을 받아온다.
- 두 번째 이후 PC 온보딩: 완전히 빈 `~/.namu`(클론이 아닌 새 설치)에서 같은 `namu_sync_setup`을 한 번 부르면, 로컬 초기 커밋과 원격의 기존 역사가 unrelated-histories라도 자동 병합(`--allow-unrelated-histories` + `merge=union`으로 `learnings.yaml` 충돌 없이 흡수)한 뒤 push한다 — 기존 원격 기억을 잃지 않는다.
- 실패(오프라인·인증 만료 등)는 세션 시작·`namu_record` 자체를 절대 막지 않고 무음 처리된다. 대신 사유 1줄이 `~/.namu/db/sync.log`에 남는다 — 기록은 되는데 원격에 안 보이면 이 로그부터 확인한다. 원인을 고치면 다음 `namu_record` 때 밀려 있던 로컬 커밋까지 한꺼번에 push된다.
- 끄기 스위치: 환경변수 `NAMU_SYNC=0`.

### 켜지 않는 경우 / clone 개발형

명시적으로 켜지 않으면 위 로컬 한계가 그대로다. clone 개발형에서도 데이터 루트는 설치형과 똑같이 `~/.namu` 고정이므로(namu-35), `namu_sync_setup`을 켜면 동일하게 동작한다 — 예전에 있던 "clone 개발형은 거부" 하드가드는 데이터 루트가 repo를 가리킬 수 있는 경로 자체가 사라지면서 함께 삭제됐다.

**"완전 무인 백업"은 아니다.** 원격 저장소 준비·git 인증은 여전히 사용자 책임이며, 인증이 깨지면 조용히 로컬에만 기억이 쌓인다 — 위 `sync.log` 확인이 유일한 단서다.

## 뺀 것 (소재에 실측 근거가 없어 제외)

- agy 설치형에서의 첫 하루 — 이 문서의 실측은 Claude Code 설치형만 다뤘다. agy 쪽 설치·재설치는 install_guide 5절 참고.
- 워커 엔진을 native 이외로 바꾸는 사용법 — 미구현(install_guide 7절과 동일 사유).
- statusLine 자동 등록 — plugin.json 메커니즘이 없어 여전히 수동 설정이며, 5절 안내 이상의 자동화는 없다.
