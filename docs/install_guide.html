1. 이게 뭔가요?

NAMU는 AI 비서(Claude Code, agy 같은 프로그램)에 끼워 넣는 "기억 장치"라고 생각하시면 쉽습니다. 원래 AI 비서는 대화가 끝나면 그 내용을 대부분 잊어버립니다. NAMU를 설치하면, AI가 작업하면서 얻은 교훈과 진행 중이던 작업 상태를 옆에 붙은 기억 장치에 계속 쌓아 두고, 다음에 다시 켰을 때도 그 기억을 불러와 씁니다.

플러그인(plugin)이란, 프로그램 본체를 건드리지 않고 나중에 끼워 넣을 수 있는 부품을 말합니다. NAMU도 Claude Code나 agy에 "설치"하는 방식으로 끼워 넣습니다.

설치형 vs clone 개발형 vs 셀프호스팅형

NAMU를 쓰는 방법은 세 가지입니다. 이 문서는 ①번, 설치형을 다룹니다.

  • 설치형 — NAMU repo와 무관한 내 프로젝트에 NAMU를 플러그인으로 설치해서 씁니다. 설치되는 것은 원본의 복사본이며, 기억은 항상 ~/.namu라는 별도 폴더에 저장됩니다. 이 문서가 안내하는 방식입니다.
  • clone 개발형 — NAMU repo 자체를 내려받아(clone) 그 안에서 직접 개발하며 씁니다. NAMU를 직접 고치고 싶은 개발자용이며, 이 문서의 대상이 아닙니다.
  • 셀프호스팅형 — Claude Code·agy 프로그램 설치 없이 웹 브라우저의 claude.ai에서만 NAMU 기억(교훈)을 쓰고 싶을 때 씁니다. namu-plugin/http_server.py를 인터넷에 상시로 띄워 claude.ai의 Custom Connector(커스텀 커넥터)로 연결합니다. 앞의 두 형태와 달리 세션 자동 브리핑·/namu-task 같은 자동화는 동작하지 않는 등 한계가 뚜렷합니다 — 이 문서의 대상이 아니며, 절차·한계 모두 웹에서 쓰기 가이드에서 다룹니다.
기억은 어디에 저장되나요?

내 컴퓨터의 ~/.namu 폴더(사용자 홈 폴더 아래 .namu)에 저장됩니다 — 설치형이든 clone 개발형이든, 어느 프로젝트에서 쓰든 항상 이 한 곳입니다(위치는 고정이며 바꾸는 설정이 없습니다). 이 폴더는 미리 만들어 둘 필요 없이, NAMU가 처음 무언가를 기억할 때 자동으로 생깁니다.

2. 시작 전 준비물 체크리스트

아래 항목을 하나씩 확인하고 넘어가세요. 하나라도 빠지면 설치 중간에 막힐 수 있습니다.

참고

NAMU repo(onmiso-hash/namu-agent)는 공개(public) 저장소입니다. 별도의 초대나 git 자격증명 없이 누구나 바로 설치를 시작할 수 있습니다.

3. 설치하기 — Claude Code (권장)

가장 간단한 방법은 GitHub 저장소를 원격 마켓플레이스(플러그인을 모아 둔 등록소)로 등록하고 바로 설치하는 것입니다. 명령 두 줄이면 끝납니다.

1

마켓플레이스 등록

터미널(명령을 입력하는 검은 화면)에 아래 명령을 입력하고 엔터를 누릅니다.

claude plugin marketplace add onmiso-hash/namu-agent
2

NAMU 플러그인 설치

이어서 아래 명령을 입력합니다.

claude plugin install namu@namu-marketplace

Claude Code 대화 세션(채팅 창) 안에서라면 /plugin marketplace add onmiso-hash/namu-agent로 시작해도 동일하게 동작합니다.

참고

repo가 공개(public)이므로 별도 초대나 git 자격증명 없이 바로 이 단계가 성공합니다. 설치 범위는 기본적으로 "이 컴퓨터를 쓰는 나"(user 스코프)입니다.

내 컴퓨터에 이미 내려받은 폴더로 설치하고 싶다면 (로컬 directory 방식)

NAMU repo를 이미 clone해 뒀거나, 개발자에게서 namu-plugin 폴더를 직접 전달받은 경우 이 방식을 쓸 수 있습니다. Claude Code 대화 세션 안에서 아래 세 줄을 순서대로 입력합니다. /path/to/namu-plugin 자리에는 실제 namu-plugin 폴더의 경로를 넣습니다. 

/plugin marketplace add /path/to/namu-plugin
/plugin marketplace update
/reload-plugins

이 방식은 폴더의 내용을 실시간으로 그대로 참조합니다. 그래서 나중에 그 폴더의 내용이 최신화되면(예: git pull), Claude Code를 다시 시작하기만 해도 반영됩니다 — 재설치가 필요 없습니다.

로컬 폴더 방식으로 설치했다가 나중에 GitHub 원격 방식으로 바꾸고 싶다면

아래 세 줄을 순서대로 실행하면 바꿀 수 있습니다(2026-07-13 실측 확인). 

claude plugin marketplace remove namu-marketplace
claude plugin marketplace add onmiso-hash/namu-agent
claude plugin install namu@namu-marketplace

내부 설정 파일(settings.json)의 등록 정보는 두 번째 명령이 알아서 로컬 폴더 표시에서 GitHub 표시로 바꿔 주므로 직접 수정할 필요는 없습니다. 다만 이렇게 바꾼 다음부터는 더 이상 폴더를 실시간 참조하지 않고 "복사본을 설치하는" 방식으로 바뀌기 때문에, 이후 반영은 git pull이 아니라 아래 8절의 업데이트 명령을 써야 합니다. 로컬 폴더 방식과 GitHub 원격 방식이 이름이 같은 등록 창구를 같이 쓰기 때문에, 반대로 실수하면(원격 등록을 다시 하면) 개발용 로컬 폴더 설정이 조용히 GitHub 쪽으로 바뀌어 버릴 수 있으니 개발 중인 컴퓨터에서는 주의하세요.

4. 설치하기 — agy

GitHub 저장소 주소를 바로 넘기면 agy가 clone·설치를 대신 처리합니다. 로컬에 namu-plugin 폴더를 미리 마련할 필요가 없습니다. 

agy plugin install https://github.com/onmiso-hash/namu-agent.git
이미 로컬에 내려받은 namu-plugin 폴더가 있다면 (로컬 경로 방식)

이 repo를 이미 clone해 두었거나 개발자에게서 namu-plugin 폴더를 직접 전달받은 경우 아래 명령을 씁니다. <namu-plugin 경로> 자리에는 실제 폴더 경로를 넣습니다. 

agy plugin install <namu-plugin 경로>

더 편하고 안전한 방법은, 설치부터 재설치·업데이트까지 한 번에 처리해 주는 스크립트를 쓰는 것입니다(Windows PowerShell 전용). 터미널에서 NAMU repo(또는 namu-plugin 폴더가 있는 위치) 기준으로 아래를 실행하세요. 

namu-plugin/scripts/agy_reinstall.ps1

이 스크립트는 기존 설치를 먼저 깨끗이 지운 뒤(agy의 설치 방식은 지운 파일도 남기는 "병합" 방식이라, 완전히 새로 하려면 먼저 지워야 합니다) 다시 설치하고, 경로 설정까지 즉시 손봐 줍니다.

안심하세요

재설치 직후 첫 세션에서는 /mcp 상태가 조금 이상하게 보일 수 있습니다. 이건 정상입니다. agy가 실행 시점에 경로를 자동으로 손보는 절차가 아직 한 번도 돌지 않아서 그런 것이고, 잠시 후 대화창에서 /namu:update(8절 참고)를 한 번 부르면 경로 교정까지 즉시 처리됩니다.

5. 잘 설치됐는지 확인하기

Claude Code

1

터미널에 아래 명령을 입력합니다.

claude mcp list

목록에 plugin:namu:namu-memory 항목이 Connected(연결됨) 상태로 보이면 성공입니다.

2

새 대화 세션을 열어 화면 하단 상태줄에 NAMU 관련 정보가 표시되는지, 대화창에서 /mcp로 도구 목록을 열었을 때 namu_record · namu_recall · namu_search 3개의 도구가 보이는지 확인하세요.

agy

새 세션을 열고 /mcp 상태를 확인합니다. 앞서 안내한 대로, 재설치 직후 첫 세션에서만 이상하게 보일 수 있고 그다음 세션부터는 정상으로 보여야 합니다.

6. 선택: 교훈 원격 백업·멀티 PC 동기화

지금까지의 설치만으로도 NAMU는 잘 동작합니다. 다만 ~/.namu에 쌓이는 기억(교훈)은 기본적으로 그 컴퓨터의 로컬 폴더일 뿐이라, 디스크가 고장 나면 기억도 사라지고 컴퓨터를 두 대 이상 쓰면 기억이 기기마다 따로 쌓입니다. 이 절은 그 한계를 없애는 선택 기능을 안내합니다 (버전 0.1.11부터, 아래 절차로 설정해야만 켜집니다).

1

비공개 원격 저장소 준비 (내 몫)

"준비"의 범위는 딱 세 가지입니다 — 여기까지만 내 몫이고, 나머지는 다음 단계에서 NAMU가 대신합니다.
① 계정 — GitHub 기준으로 설명하지만, GitLab·사내 git 서버 등 git 주소가 나오는 곳이면 어디든 됩니다.
② 빈 비공개 저장소 — New repository → 이름 자유(예: namu-memory) → Private 선택 → README 등 초기화 옵션은 체크하지 않고(빈 저장소 권장) 생성합니다. 비공개여야 하는 이유: 교훈에는 여러분이 작업한 프로젝트의 내용·판단·실수가 그대로 담기기 때문입니다. GitHub은 비공개 저장소도 무료입니다.
③ 이 컴퓨터의 git 로그인 — 터미널에서 push할 때 비밀번호를 묻지 않는 상태여야 합니다. gh auth login(가장 간단) 또는 SSH 키 등록으로 준비하고, gh auth status로 확인할 수 있습니다. 이 컴퓨터에서 GitHub에 push해 본 적이 있다면 대개 이미 돼 있습니다.

2

AI에게 연결을 한 번만 부탁하기

Claude Code나 agy 대화창에서 그냥 이렇게 말하면 됩니다.

namu_sync_setup을 <위에서 준비한 원격 저장소 주소>로 실행해줘

AI가 namu_sync_setup MCP 도구를 호출해 로컬 ~/.namu에 git 연결을 자동으로 준비합니다(초기화·원격 등록·설정 파일 작성·첫 백업까지). 한 번만 하면 끝입니다.

켠 뒤에는 무슨 일이 일어나나요?

교훈을 기록할 때마다(namu_record) 자동으로 백업(push)되고, Claude Code나 agy로 새 세션을 열 때마다 자동으로 다른 컴퓨터의 최신 기억을 받아옵니다(pull). 두 번째 컴퓨터에서도 위 1·2번 절차를 그대로 반복하면, 그 컴퓨터가 완전히 새 것이라도(클론이 아니어도) 원격에 있는 기존 기억과 자동으로 합쳐집니다 — 따로 병합 작업을 할 필요가 없습니다.

알아둘 점

"완전 무인 백업"은 아닙니다 — 원격 저장소 준비와 git 로그인은 여전히 사용자 몫이고, 로그인이 끊기면 조용히 실패하며 기억은 그 컴퓨터에만 계속 쌓입니다(겉으로 오류가 뜨지 않습니다). 문제가 있는지 확인하려면 ~/.namu/db/sync.log 파일을 열어보세요 — 실패 사유가 한 줄씩 남습니다. 끄고 싶다면 환경변수 NAMU_SYNC=0을 설정하면 됩니다. 참고로 NAMU repo 자체를 clone해 개발하는 방식(이 문서의 대상이 아닌 방식)에서는 이 기능이 안전을 위해 자동으로 비활성화됩니다.

7. 삭제할 때

플러그인을 지우는 방법도 어떻게 설치했는지에 따라 다릅니다.

Claude Code — GitHub 원격으로 설치한 경우

claude plugin uninstall namu@namu-marketplace

짧은 이름(uninstall namu)이 아니라 이름@마켓플레이스 전체 이름을 쓰는 것이 안전합니다 — 아래 8절 업데이트와 같은 규칙입니다. 마켓플레이스 등록 자체까지 지우고 싶다면 아래도 실행하세요. 

claude plugin marketplace remove namu-marketplace

Claude Code — 로컬 폴더(directory)로 설치한 경우

위 두 명령에 더해, 설정 파일(settings.json)의 마켓플레이스 등록 목록에 로컬 폴더 경로가 그대로 남아 있으면 그것도 지워야 완전히 삭제됩니다. 안 지우면 Claude Code를 다시 시작할 때 로컬 폴더 소스가 다시 등록돼 버립니다(2026-07-13 실측 확인).

agy

agy plugin uninstall namu

agy의 plugin install은 지운 파일도 그대로 남기는 "병합" 방식이므로(4절·9절 참고), 완전히 깨끗하게 지우려면 반드시 uninstall을 먼저 해야 합니다. agy_reinstall.ps1 스크립트도 내부적으로 이 순서(uninstall → install → --heal)를 그대로 따릅니다.

기억은 지워지지 않아요

플러그인을 삭제해도 ~/.namu에 쌓인 교훈·작업 상태는 그대로 남습니다 — 기억이 저장되는 위치가 플러그인 설치 위치와 완전히 별개이기 때문입니다. 나중에 다시 설치하면 기억이 그대로 돌아옵니다. 기억까지 완전히 지우고 싶다면 ~/.namu 폴더를 따로 지워야 합니다.

8. 업데이트할 때

삭제하고 다시 설치하는 방법은 위 7절을 참고하세요.

원클릭 — /namu:update (기본 절차)

대화창에서 아래처럼 말하면 끝입니다.

/namu:update

설치된 호스트(Claude Code·agy)를 자동으로 감지해 각각 알맞은 방식으로 갱신하고, 끝나면 /namu:statusline-setup까지 자동으로 호출해 statusLine 경로를 새 버전으로 다시 연결합니다. 미설치된 호스트는 조용히 건너뜁니다. 완료 후 갱신된 호스트를 재시작하면 반영됩니다.

스킬을 쓸 수 없는 환경 — 수동 업데이트 절차

NAMU 코드가 새로 바뀌었을 때 수동으로 반영하는 방법은 어떻게 설치했는지에 따라 다릅니다.

설치 방식 Claude Code agy
로컬 폴더(directory) 방식 git pull(또는 소스 폴더 최신화) 후 Claude Code 재시작만 하면 됩니다. 재설치는 필요 없습니다. namu-plugin/scripts/agy_reinstall.ps1 실행
GitHub 원격 설치 
claude plugin update namu@namu-marketplace
실행 후 Claude Code 재시작. 이름@마켓플레이스 전체 이름을 꼭 써야 합니다 — 짧게 update namu만 쓰면 "찾을 수 없음(not found)" 오류가 납니다. 		namu-plugin/scripts/agy_reinstall.ps1 실행

기억해 두면 좋은 구분: 로컬 폴더 방식은 원본 폴더를 "실시간으로" 그대로 보고 실행하는 방식이라 폴더만 최신화하면 끝나지만, GitHub 원격 설치는 설치 시점의 내용을 "복사"해서 실행하는 방식이라 명시적으로 업데이트 명령을 내려야 반영됩니다.

업데이트 후 statusLine 확인 — 화면 하단 statusLine이 연결된 설치본(캐시) 경로에는 플러그인 버전이 들어 있어서, 업데이트로 버전이 올라가면 statusLine이 오류 표시 없이 조용히 사라집니다. 업데이트 직후 대화창에서 /namu:statusline-setup을 다시 호출하면 새 버전 경로로 자동 갱신됩니다 (0.1.16부터 기본 절차, 사용설명서 usage_guide.md 6절 참고). 이 스킬을 쓸 수 없는 환경이라면 settings.json의 statusLine 경로 속 버전을 수동으로 바꿔 주세요(같은 절의 폴백 절차 참고). 위 /namu:update 원클릭을 썼다면 이 확인 자체가 이미 자동으로 처리된 것이므로 신경 쓸 필요 없습니다.

9. 문제가 생겼어요

자주 겪는 증상을 모았습니다. 해당하는 항목을 눌러 펼쳐 보세요.

세션을 시작해도 이전 작업 상태나 교훈이 안 떠요
원인 Windows 환경에서 세션이 시작될 때 저장된 내용을 자동으로 불러오는 과정이 한글·이모지가 섞인 내용을 화면에 표시하려다 인코딩 문제로 조용히 실패하는 오래된 버그입니다(버전 0.1.5 이하에서 발생). 이 실패가 겉으로 오류 없이 조용히 일어나기 때문에, 그냥 "안 뜨네" 정도로만 보입니다.
해결 NAMU를 0.1.6 이상으로 업데이트하세요(위 8절 참고). 이 버전부터 해당 문제가 해소됐습니다.
업데이트했는데 반영이 안 돼요
원인 /namu:update를 안 쓰고 수동 절차를 밟은 경우, 설치 방식에 따라 "업데이트"의 의미가 다릅니다. 로컬 폴더 방식은 원본을 실시간으로 참조하지만, GitHub 원격 설치는 설치 당시 내용을 복사해 둔 것을 실행하는 방식이라, 원본만 최신화해서는 복사본이 갱신되지 않습니다.
해결 가장 쉬운 방법은 대화창에서 /namu:update를 다시 부르는 것입니다. 수동으로 하려면 8절 표에서 내 설치 방식에 맞는 절차를 따르세요. 특히 GitHub 원격 설치는 claude plugin update namu@namu-marketplace처럼 이름@마켓플레이스 전체 이름을 반드시 써야 합니다.
agy에서 /mcp 상태가 이상해요
원인 agy는 실행할 때마다 필요한 경로를 자동으로 손보는데, 재설치 직후 첫 세션에서는 이 손보는 과정이 아직 한 번도 돌지 않아 일시적으로 이상하게 보일 수 있습니다. 정상적인 현상입니다.
해결 namu-plugin/scripts/agy_reinstall.ps1을 실행하면 설치 직후 바로 경로를 교정해 주므로 첫 세션부터 정상으로 보입니다. 이미 실행했는데도 계속 이상하다면 스크립트를 다시 한번 실행해 보세요.
agy에서 자동화 스크립트(비대화 모드)로 NAMU를 쓰고 싶어요
원인 agy의 비대화(자동, -p) 모드는 NAMU 같은 MCP(메모리) 도구가 설정돼 있으면 세션 전체가 멈춰 버리는 현재 알려진 제한입니다.
해결 아직 방법이 없습니다. agy에서 NAMU는 대화형 세션에서만 정상 동작합니다. 자동화가 꼭 필요하다면 Claude Code 쪽을 이용해 주세요.
재설치했는데 지웠던 파일이 그대로 남아 있어요 (agy)
원인 agy의 plugin install은 기존 설치 위에 덧씌우는 방식이라, 원본에서 삭제된 파일이 설치본에는 그대로 남습니다.
해결 반드시 먼저 지운(uninstall) 뒤 다시 설치해야 완전히 깨끗해집니다. namu-plugin/scripts/agy_reinstall.ps1이 이 순서를 자동으로 처리해 주니, 이 스크립트를 쓰는 것이 가장 안전합니다.
마켓플레이스를 다시 등록했더니 다른 소스로 바뀌어 버렸어요
원인 이미 등록된 것과 같은 이름(namu-marketplace)으로 다시 marketplace add를 실행하면, 오류 없이 조용히 기존 등록이 새 소스로 교체됩니다.
해결 같은 이름으로 두 번 등록하지 않도록 주의하세요. 실수로 바뀌었다면 원하는 방식(원격 또는 로컬 폴더)으로 다시 marketplace add를 실행해 원하는 소스로 되돌리면 됩니다.

10. 더 알아보기

NAMU가 실제 작업을 시킬 때 부르는 일꾼(워커, 서브에이전트)이 있습니다. 특별한 설정 없이도 기본값으로 잘 동작하므로, 대부분의 설치형 사용자는 아래 내용을 몰라도 됩니다.

  • 워커 설정 파일(namu_workers.yaml)이 없으면 — 지금 쓰고 있는 프로그램(Claude Code 또는 agy) 자체의 기본 서브에이전트를 그대로 씁니다. 추가 설정도 추가 비용도 없습니다.
  • 이 파일이 있으면 — 워커마다 어떤 엔진을 쓸지 지정할 수 있습니다. 다만 지금 실제로 동작하는 값은 native(현재 프로그램의 기본 서브에이전트)뿐입니다.
  • 워커가 실제로 쓰는 AI 모델을 바꾸고 싶다면, 이 yaml 파일이 아니라 서브에이전트 정의 파일 안의 model: 값을 고쳐야 합니다.

더 자세한 내용(개발 배경, 실측 기록 포함)은 install_guide.md를 참고하세요. 웹 브라우저에서만 NAMU를 쓰고 싶다면 웹에서 쓰기 가이드를 참고하세요.
