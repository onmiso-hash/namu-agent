1NAMU가 뭐예요?

한 문장으로: "AI는 갈아끼워도, 기억은 내가 들고 다니는 시스템"

NAMU(나무)는 클로드·제미나이·GPT 같은 여러 AI를 부려서 일을 시키되, 그 과정에서 얻은 경험과 교훈을 한 곳에 차곡차곡 쌓아두는 개인용 AI 작업 시스템이에요. 핵심은 "어떤 AI를 쓰느냐"가 아니라 "그 기억을 누가 가지고 있느냐"예요.

💡 쉬운 비유

보통 AI 챗봇은 그 회사 앱을 나가면 대화 내용을 다 잊어버려요. 마치 매번 처음 만나는 직원 같죠. NAMU는 그 직원이 바뀌어도 똑같이 보는 공용 업무 노트를 내가 직접 들고 있는 구조예요. 직원(AI)은 바뀌어도 노트(기억)는 그대로라서, 새 직원도 노트만 읽으면 바로 일을 이어받아요.

"독립성은 화면(인터페이스)이 아니라
기억(메모리)에 있다"
이게 NAMU 프로젝트 전체를 관통하는 가장 중요한 깨달음이에요

2왜 만들어요? — 기획 의도

세 가지 바람에서 출발했어요

🔓

① AI에 안 묶이기

클로드가 막히면 제미나이로, 그것도 안 되면 내 PC의 무료 모델로. 한 회사 정책·요금·서비스 종료에 휘둘리지 않아요.

🧠

② 기억 공유하기

모든 AI가 같은 노트를 봐요. 게다가 노트북·집 PC 어디서 작업해도 같은 기억이 따라다녀요.

📈

③ 스스로 발전하기

성공·실패만이 아니라 "왜 그렇게 판단했는지 이유"까지 쌓아요. 이게 쌓이면 시스템이 스스로 기준을 만들어가요.

🌍

④ (먼 미래) 함께 크기

나중엔 일반적인 교훈을 커뮤니티 공용 풀에 선택적으로 나눠서, 쓸수록 모두가 똑똑해지는 구조도 꿈꿔요.

"이유까지 기록한다"가 왜 중요할까?

그냥 "성공/실패"만 적으면, 나중에 패턴을 분석할 때 엉뚱한 기준을 뽑아낼 수 있어요. 예를 들어 "금요일에 한 작업은 다 실패했네 → 금요일을 피하자" 같은 헛다리요. 하지만 왜 실패했는지(이유)가 같이 적혀 있으면 "아, 테스트를 안 돌려서 실패한 거구나"라는 진짜 원인을 잡을 수 있어요. 그래서 NAMU는 reason(이유)을 반드시 적도록 강제해요. 이유 없이는 기록 자체가 거부돼요.

3어떻게 만들어요? — 설계

"다 직접 만들면 끝이 없다" → 빌릴 건 빌리고, 핵심만 직접 만들어요

처음엔 모든 걸 직접 만들려다가, 중요한 결정을 했어요. AI 작업 시스템을 3개 층으로 나눠서, 이미 잘 만들어진 부분은 가져다 쓰고 진짜 핵심인 기억 부분만 직접 만들기로요.

A

실행 엔진

AI를 호출하고, 파일을 고치고, 명령어를 돌리는 부분. (예: Claude Code, agy=안티그래비티)

빌려쓴다
B

하네스 엔지니어링

"작업 전엔 꼭 기억을 확인해라" 같은 규칙·자동화·작업 절차. NAMU가 직접 만들어요.

직접 제작
C

메모리 / 학습 코어 💚

경험을 쌓고 꺼내쓰는 부분. 이게 NAMU의 심장이자 진짜 차별점이에요.

직접 제작
왜 이렇게 나눴나요? 실행 엔진(A)은 큰 회사들이 수년간 다듬은 부분이라 직접 만들면 시간 낭비예요. 반면 "기억과 학습"(C)은 어디서도 안 주는, NAMU만의 가치라서 여기에 힘을 집중해요.

핵심 코어(C층)는 'MCP 서버'라는 형태로

기억 코어를 MCP 서버라는 형태로 만들었어요. MCP는 쉽게 말해 "AI에 갖다 붙이는 표준 콘센트"예요. 이 콘센트 규격으로 만들어두면 클로드든, 다른 AI 도구든 똑같이 꽂아서 같은 기억을 쓸 수 있어요. 바로 이 점이 "AI에 안 묶인다"는 목표를 진짜로 실현해줘요.

제미나이도 지원돼요 — 이제는 완료됐어요. 예전에 포기했던 건 "NAMU가 agy(안티그래비티)를 직접 조종하던 옛 구독 어댑터" 딱 하나였어요(non-TTY 출력 버그 때문). 대신 agy를 실행엔진(A층)으로 빌려쓰고 그 위에 같은 NAMU 메모리(MCP)를 붙이는 '글루'를 완성했어요. 그래서 지금은 클로드 코드와 똑같은 방식으로 agy(제미나이)에서도 같은 기억을 써요. 다만 이건 "완전 지원"이라기보다, 같은 플러그인을 설치해 같은 기억 코어에 꽂았다는 뜻이에요.

learnings.yaml = 진짜 노트

사람이 그냥 읽을 수 있는 텍스트 파일. 추가만 가능(삭제·수정 금지)해서 이력이 보존돼요. 깃(GitHub)으로 여러 PC가 이 노트를 공유해요. 이게 진실의 원천.

🔍

SQLite = 빠른 색인

노트를 검색이 잘 되게 정리해둔 데이터베이스. 어디까지나 캐시라서, 망가져도 노트(yaml)로 언제든 다시 만들 수 있어요.

왜 둘로 나눴나요? "노트 먼저 쓰고, 색인은 나중에" 순서가 핵심이에요. 만약 색인(DB)이 깨져도 노트만 멀쩡하면 전부 복구할 수 있거든요. 그래서 기록할 때 항상 yaml에 먼저 저장해요. 이 순서는 절대 안 바꿔요.

어디에 저장되나 — 같은 중앙 폴더(~/.namu) 안에서 두 갈래로

🌍

교훈(learnings.yaml)+SQLite → ~/.namu

어느 프로젝트에서 일하든 공통으로 써먹을 지식이라, 프로젝트 구분 없이 한 풀에 섞어서 기기 전역 폴더(~/.namu)에 모여요.

📁

작업 상태(tasks) → ~/.namu/tasks/<프로젝트폴더명>/

지금 이 프로젝트에서만 의미 있는 상태지만, 저장 위치는 교훈과 같은 중앙 ~/.namu 안에 프로젝트 폴더명별로 칸을 나눠 담겨요. 프로젝트 폴더(공개 repo일 수 있어요)에 개인 작업기록이 딸려 나가는 걸 막고, 여러 PC 간 자동 동기화에 같이 얹으려고요.

💡 쉬운 비유

둘 다 내 중앙 서랍장(~/.namu)에 살아요. 다만 일반 상식(교훈)은 프로젝트를 안 가리고 한 칸에 다 섞어 모아두고(어디서든 꺼내 쓰려고), 이번 프로젝트 메모(작업 상태)는 같은 서랍장 안에서 프로젝트별 칸으로 나눠 넣어둬요. 예전엔 작업 상태를 각 프로젝트 폴더에 뒀지만, 공개 repo에 개인 기록이 새는 걸 막고 PC 간 자동 공유에 얹으려고 중앙으로 옮겼어요.

기억을 다루는 도구 3개

AI가 이 콘센트(MCP)에 꽂으면 쓸 수 있는 도구가 딱 세 개예요. 단순하게 유지하는 게 목표였어요.

namu_recall기억 꺼내기

작업을 시작하기 전에 "예전에 비슷한 거 해봤나?" 하고 관련 기억을 불러와요. 딱 맞는 게 없으면 최근 것이라도 몇 개 보여줘서 뭐라도 맥락을 잡게 해줘요.

namu_record기억 저장하기

작업이 끝나면 결과(성공/실패/부분성공)와 판단 이유를 함께 기록해요. 이유가 비어 있으면 저장이 거부돼요 — 이게 자가학습의 핵심 규칙이에요.

namu_search패턴 찾기

쌓인 기록에서 비슷한 사례·패턴을 분석적으로 찾아요. 정확히 맞는 것만 보여주고, "성공 N건·실패 M건" 같은 경향 요약도 같이 붙여줘요.

한글 검색도 신경 썼어요. 일반 검색 방식은 한글 부분검색이 잘 안 돼서, trigram이라는 방식(3글자 단위로 쪼개 검색)을 썼고, 2글자 이하 짧은 검색어는 별도 방식으로 보완했어요.

5지금까지 한 일

기획 → 설계 → 기억 코어 완성 → 자동화 → 패키징 → 멀티 PC 실측까지 왔어요

✓

기억 코어 완성 완료

"자체 CLI 직접 제작" 대신 "기억 코어(MCP) + 빌려쓴 실행엔진"으로 방향을 잡고(어댑터 5종 시행착오 포함), 도구 3개(recall·record·search)를 전부 구현·검증했어요. yaml 먼저 쓰고 SQLite 색인을 만드는 구조, 한글 검색, 이유 필수 규칙까지 동작 확인. NAMU가 개발 중 얻은 교훈을 스스로 기록하기 시작한 것도 이 단계예요.

✓

세션 자동 브리핑(훅 자동화) 완료

Claude Code 세션을 열면 진행 중인 작업·최근 교훈이 AI에게 자동 주입돼요. git이 최신이 아니면 경고(behind 감지)하고, 검색 캐시(SQLite)가 낡았으면 자동으로 다시 만들어요.

✓

/namu-task 작업 절차 완료

작업 정의(사용자 확인) → 코더 워커 구현 → 리뷰어 워커 검수 → 검수 실패 시 자동 재실행 금지(사용자에게 물음) → 교훈 기록까지 표준 절차로 완성했어요. 워커는 namu-coder/namu-reviewer 서브에이전트예요.

✓

플러그인 패키징(현재 0.1.25) + agy용 글루 완료

한 번에 설치되는 Claude Code 플러그인으로 묶었어요. 같은 플러그인을 agy(안티그래비티)에 설치하면 제미나이에서도 같은 기억을 써요.

✓

git pull 후 SQLite 자동 재생성 완료

다른 PC의 노트(yaml)를 받아오면, 검색 색인(SQLite)이 항목 수 불일치를 감지해 서버 부팅 시 자동으로 다시 만들어져요.

✓

멀티 PC 동기화 실측 완료

삼성·hp 두 기기에서 실제 작업으로 기억이 잘 따라오는지 확인했어요. 이론이 아니라 실측으로 검증.

✓

statusLine 원클릭 연결 + 원클릭 업데이트 완료

화면 하단에 [Namu 버전] [모델] 폴더 | 📌 진행 task | 컨텍스트% · 5h(리셋까지 남은 시간) · 7d(리셋까지 남은 시간)를 보여주는 statusLine이 플러그인에 함께 들어와요. Claude Code 사양상 자동으로 등록되지는 않지만, 대화창에서 /namu:statusline-setup 한 번이면 Claude Code·agy 양쪽을 자동 감지해 연결해줘요. 플러그인 코드가 새로 바뀌었을 때도 /namu:update 한 번이면 설치된 호스트를 전부 최신 버전으로 올리고 statusLine 경로까지 자동으로 다시 연결해줘요.

✓

웹 claude.ai 연결(원격 MCP) 완료

Claude Code·agy 없이 웹 브라우저의 claude.ai에서도 NAMU 기억을 쓸 수 있게 됐어요. http_server.py를 인터넷에 상시로 띄워 claude.ai의 Custom Connector(커스텀 커넥터)로 연결하는 방식이에요. 사용자 소유 상시 서버로 전 구간(서버 배포 → 고정 주소 터널 → 웹 연결 → 실제 기록)을 실측 검증했어요. 다만 웹에서는 교훈 recall·record·search 3가지만 되고, 세션 브리핑이나 /namu-task 같은 자동화는 여전히 CC/agy 몫이에요 — 자세한 절차와 한계는 웹에서 쓰기 가이드를 보세요.

6앞으로 할 일

기억 코어와 자동화, 그리고 설치형 기억의 원격 백업까지 자리를 잡았어요. 이제 남은 건 "워커 다양화"와 "다듬기"예요

✓

설치형 기억의 자동 백업·멀티 PC 동기화 완료

설치형(~/.namu)에 namu_sync_setup 도구로 원격 저장소를 한 번만 연결해두면, 그 뒤로는 교훈을 기록할 때마다 자동 백업(push)되고 새 세션을 열 때마다 자동으로 다른 PC의 기억을 받아와요(pull). 두 번째 PC도 같은 명령 한 번이면 기존 기억과 자동으로 합쳐져요. 연결해두지 않으면 여전히 그 기기의 로컬 폴더일 뿐이에요 — 원격 저장소 준비와 git 로그인은 여전히 내 몫이고, 로그인이 끊기면 조용히 로컬에만 쌓이는 점은 알아둬야 해요.

🔄 동기화 켜는 법
① GitHub 같은 곳에 빈 비공개 저장소를 하나 만들고(README 없이 생성, 비공개도 무료예요), 그 컴퓨터에서 git 로그인(gh auth login 등)까지 해둬요 — 여기까지가 내 몫이에요 → ② AI에게 "namu_sync_setup을 이 주소로 실행해줘"라고 한 번만 부탁하면 끝이에요 → ③ 그다음부턴 교훈을 기록할 때마다 자동 백업(push), 새 세션을 열 때마다 자동 수신(pull)이 이뤄져요 → ④ 두 번째 PC에서도 같은 명령 한 번이면 기존 기억과 자동으로 합쳐져요(따로 병합 작업 필요 없음).

1

마켓플레이스 등록 + 설치

Claude Code에서 이 두 줄이면 NAMU의 기억 도구 3개와 서브에이전트가 통째로 설치돼요.

# Claude Code에서 실행
claude plugin marketplace add onmiso-hash/namu-agent
claude plugin install namu@namu-marketplace
참고: NAMU repo는 공개(public) 저장소예요. 별도로 초대받거나 git 자격증명을 미리 준비할 필요 없이, 누구나 위 두 줄만으로 바로 설치할 수 있어요.
📘 자세한 설치·문제해결은 설치 가이드에서
로컬 경로 설치 방식, agy(안티그래비티) 설치, 업데이트 절차, 트러블슈팅까지는 이 안내서 범위를 넘어요. → install_guide.html 보러 가기
2

설치 확인

Claude Code에서 /mcp를 치면 plugin:namu:namu-memory ✔ connected · 3 tools가 보여야 정상이에요.

첫 세션은 조용한 게 정상이에요. 설치 직후엔 보여줄 기억(교훈·진행 중 작업)이 아직 하나도 없어서, 환영 메시지도 브리핑도 안 떠요. 실패가 아니라 기억이 쌓인 뒤 다음 세션부터 브리핑이 시작되는 거예요.

기기 이름(NAMU_MACHINE)은 이제 설정 안 해도 컴퓨터 호스트명으로 자동 인식돼요. 필수 설정이 아니고, 여러 PC를 쓸 때 라벨을 예쁘게 고정하고 싶을 때만 정해주면 돼요.

🔄 ② 매번 하는 작업 흐름

💻

설치형 — 대부분의 사용자

그냥 내 프로젝트에서 Claude Code를 열면 돼요. 세션 브리핑이 자동으로 주입되고, /namu-task로 작업을 진행하고, 일반화할 교훈이 있으면 AI가 판단해 namu_record로 남겨요(이유 필수). 교훈은 ~/.namu에 자동 저장되니 따로 pull·push할 필요가 없어요 — 원격 백업·멀티 PC 동기화까지 자동으로 하고 싶으면 켜는 방법이 따로 있고(6장 참고), 켜지 않으면 예전처럼 그 폴더의 백업은 스스로 챙겨야 해요.

🛠️

clone 개발형 — NAMU 저장소 자체를 개발할 때만

NAMU 코드 자체를 고치는 개발자라면 그 repo는 받아오기(pull) → 작업 → 올리기(push)로 코드를 관리해요. 하지만 기억 저장 위치는 설치형과 똑같이 ~/.namu예요(저장소 안이 아니에요) — 그래서 교훈·작업 상태의 백업·동기화는 위 설치형과 완전히 같은 방식이에요. 일반 사용자는 해당하지 않아요.

🌐

셀프호스팅형 — Claude Code/agy 없이 웹 브라우저에서만

설치도, 프로그램 실행도 없이 웹 브라우저의 claude.ai에서만 NAMU 기억을 쓰고 싶다면 이 형태예요. NAMU의 기억 서버(http_server.py)를 인터넷에 상시로 띄워 claude.ai의 Custom Connector(커스텀 커넥터)로 연결해요. 대신 recall·record·search 3가지만 되고, 세션 브리핑·/namu-task 같은 자동화는 없어요 — 절차·한계 모두 → remote_mcp_guide.html 보러 가기

💡 저장 위치, 한 줄 정리

교훈도 작업 상태(tasks)도 모두 중앙 ~/.namu에 담겨요 — 교훈은 프로젝트 구분 없이 한 풀에, 작업 상태는 ~/.namu/tasks/<프로젝트폴더명>/처럼 프로젝트별 칸으로 나눠서요. 예전엔 작업 상태만 각 프로젝트 폴더에 뒀는데 지금은 중앙으로 통합됐어요 — 자세한 이유는 4장 참고.

기억 도구를 직접 써보고 싶다면?
· 꺼내기: "예전에 SQLite 관련 작업 기록 있어?" → AI가 namu_recall 호출
· 저장: "이번 작업 성공으로 기록해줘, 이유는 ○○" → namu_record (이유 꼭 필요!)
· 분석: "테스트 관련 작업들 성공률 어때?" → namu_search (경향 요약까지)
